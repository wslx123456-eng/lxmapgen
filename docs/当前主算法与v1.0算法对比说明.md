# 当前主算法与 v1.0 算法对比说明

## 1. 文档目的

这份文档用于把当前项目中的两套算法路线放在同一张图里看清楚：

- 你现在主推的 `GeoTIFF -> Lane/Intersection GeoJSON` 算法
- `unimapgen-v1.0` 里的旧算法，以及我为它补的当前数据适配路径

重点不是只写“代码在哪里”，而是回答下面这些更实际的问题：

- 两套算法分别在学什么
- 输入和输出到底有什么本质差别
- 为什么当前主算法更贴近你的任务目标
- 为什么 `v1.0` 更容易做成一个能跑的 baseline
- 两套算法各自的优势、短板和适用场景是什么

---

## 2. 当前主算法是什么

### 2.1 总体目标

当前主算法是围绕你的真实任务目标设计的：

- 输入是真实遥感 `tif`
- 监督是真实 `Lane.geojson` 和 `Intersection.geojson`
- 训练和推理都按 patch 进行
- 模型输出的是 patch 内的地图几何结果
- 最终希望模型理解道路与路口的局部几何关系、跨 patch 连续性，以及两者必须严丝合缝地接触

这套算法不是单纯“画线”，而是：

- 同时建模 `lane`
- 同时建模 `intersection`
- 同时处理 patch 连续性
- 同时要求拓扑对齐

### 2.2 输入形式

当前主算法的训练样本来自真实地理数据目录，例如：

- `patch_tif/0.tif`
- `patch_tif/0_edit_poly.tif`
- `label_check_crop/Lane.geojson`
- `label_check_crop/Intersection.geojson`

其中：

- `0.tif` 是遥感图像
- `0_edit_poly.tif` 是 review mask
- `Lane.geojson` 是道路线监督
- `Intersection.geojson` 是路口面监督

review mask 不是附属信息，而是监督边界的一部分：

- 只有 mask 白色区域被视为可信监督
- 黑色区域不参与有效监督

### 2.3 patch 级训练与推理

因为原始 `tif` 很大，当前主算法采用：

- 训练期滑窗切 patch
- 推理期滑窗切 patch
- patch 预测结果再拼回整图

patch 不是简单固定裁块，而是和以下因素一起决定：

- `tile_size_px`
- `overlap_px`
- `keep_margin_px`
- review mask
- 候选窗口筛选与保留区域

### 2.4 坐标表示

当前主算法的一个核心选择是：

- 大模型全程主看 **patch-local UV**
- 不主看经纬度

也就是说：

- prompt 里的几何参考是 UV
- state 里的几何参考是 UV
- target 监督文本里的几何也是 UV

最后在导出资产和最终结果时，再把 UV 转回像素/世界坐标。

这样做的动机是：

- 让模型面对稳定、局部、数值范围一致的坐标空间
- 避免经纬度数值范围过大、分布不稳定
- 让 patch 内几何关系更容易学

### 2.5 文本监督方式

当前主算法不是生成离散地图 token，而是让模型生成 **UV GeoJSON 文本**。

训练时的主要文本链是：

- `prompt_text`
- `state_text`
- `target_text`

其中：

- `prompt_text` 负责任务说明、few-shot 示例、几何约束、同 patch 参考信息
- `state_text` 负责 patch 连续性和 cut/anchor 状态传递
- `target_text` 负责当前 patch 的监督结果

训练方式是标准 teacher forcing：

- 图像前缀 + prompt + state + 真值前缀
- 预测后续 token

推理时则只有：

- 图像
- prompt
- state

然后让模型自由生成 GeoJSON 文本。

### 2.6 双任务结构

当前主算法不是单任务线条生成，而是显式双任务：

- `lane`
- `intersection`

其差别是：

- `lane` 用 `LineString`
- `intersection` 用 `Polygon`

这意味着模型不仅要学“线怎么画”，还要学：

- 路口边界怎么画
- 线与面的接触关系怎么表达

### 2.7 patch 连续性设计

当前主算法不是完整 chat history 累积，而是“压缩状态前缀”路线。

也就是：

- 每个 patch 独立成一条训练样本
- 但会带上来自相邻 patch 的 `state_text`
- state 中包含 cut/anchor 信息与局部 StateGeoJSON

这样做是为了在显存可接受的前提下保留 patch 间承接关系。

### 2.8 同 patch 伴随几何参考

这是当前主算法近期很关键的一步设计。

现在每个 patch 在训练和推理时，不只看到当前任务自身，还可以看到同 patch 的另一类几何参考：

- `lane` patch 可以看到同 patch 的 `intersection` 参考
- `intersection` patch 可以看到同 patch 的 `lane` 参考

这不是后处理修正，而是把拓扑关系提前放进模型理解阶段。

### 2.9 不参与 loss 的字段

当前主算法明确把下面这些字段从训练监督里剥离：

- `crs.name`
- `Id`
- `RoadId`

原因是：

- `crs.name` 是固定值
- `Id/RoadId` 在原始数据中常常不是稳定语义标签，而更像工程编号

所以模型不负责“记住编号”，而是专注学习几何与结构。

最终导出时再用确定性规则补回：

- `Id` 对 `lane + intersection` 共用一条递增序列
- `RoadId` 只对 `lane` 单独递增

### 2.10 核心目标

当前主算法真正想达到的不是“先生成一个差不多格式的文件”，而是：

- 学会 patch 内 lane 几何
- 学会 patch 内 intersection 几何
- 学会跨 patch 连续性
- 学会 lane 与 intersection 的局部拓扑关系
- 学会两者刚好接触，不超出也不留缝

这是一套更接近“地图理解与生成”的算法目标。

## 3. v1.0 算法是什么

### 3.1 总体目标

`unimapgen-v1.0` 的原始算法路线，更像是一套：

- 视觉编码
- 地图序列 token 化
- 自回归解码

它的主问题更接近：

- 给 patch 图像
- 生成一段“地图线序列”

这套算法更偏向一个强 baseline，而不是你当前这种更复杂的 lane/intersection 联合拓扑任务。

### 3.2 输入形式

`v1.0` 原本不是直接吃你现在这套：

- `tif`
- `Lane.geojson`
- `Intersection.geojson`

它本来依赖的是另一套数据组织，例如：

- OpenSatMap 风格 patch 数据
- family manifest
- line map 注释

所以我后来给它写了一个适配器，把你当前的数据导出成它能训练的格式。

### 3.3 输出表示

`v1.0` 的核心不是 GeoJSON 文本，而是 **结构化地图序列**。

它更像输出：

- 一组 `lines`
- 或离散化后的地图 token 序列

也就是说，它学的是一种“地图 token 语言”，而不是直接学 GeoJSON 文本语言。

### 3.4 坐标表示

`v1.0` 更偏向：

- 离散坐标
- 坐标 bin 化
- `x_i / y_i` 这种 token 化表示

它的重点是把地图几何压进一个离散序列空间里。

### 3.5 任务结构

`v1.0` 的天然强项是：

- 线图表达
- 折线序列建模

它不是按当前主算法这样明确分成：

- `lane`
- `intersection`

更不是天然围绕“线和面的严格接触”来设计的。

### 3.6 连续性设计

`v1.0` 里也有连续性思想，但更像：

- 前序 map state token 前缀
- 线序列续写

而不是当前主算法这种：

- `StateAnchorMeta`
- `StateAnchor`
- `StateGeoJSON`
- 同 patch companion 几何参考

### 3.7 当前适配器版本的 `v1.0`

我现在给 `v1.0` 做的适配器，是一条“最短可跑 baseline”路径：

- 把当前数据导出成它能接受的训练样本
- 走 LLaMAFactory / ShareGPT + images 这条现成链
- 当前适配主要针对 `Lane`

也就是说，当前适配版 `v1.0` 不是你完整目标的等价实现，而是一个：

- 方便对比
- 方便快速验证
- 方便看 baseline 表现

的最小版本。

## 4. 两套算法的核心差异

### 4.1 任务目标差异

当前主算法：

- 是真实任务导向
- 直接面对 `lane + intersection + patch continuity + 拓扑接触`

`v1.0`：

- 更偏 baseline 导向
- 更聚焦“线图序列生成”

### 4.2 输出空间差异

当前主算法：

- 输出 UV GeoJSON 文本
- 输出空间更大、更自由
- 更自然，但更难学

`v1.0`：

- 输出离散化地图序列
- 输出空间更窄
- 更容易学，但表达更受限

### 4.3 坐标表示差异

当前主算法：

- 连续 UV
- patch-local
- 保留连续几何信息

`v1.0`：

- 离散 token/bin
- 更适合稳定训练
- 但会带来量化表达的限制

### 4.4 拓扑建模差异

当前主算法：

- 明确建模 `lane` 和 `intersection`
- 试图把两者关系放进模型理解
- 可以把“必须精确接触”作为生成目标的一部分

`v1.0`：

- 更偏单类线序列
- 没有天然围绕“线-面拓扑”设计

### 4.5 连续性建模差异

当前主算法：

- `state_text`
- `cut/anchor`
- companion 几何参考

`v1.0`：

- map token state prefix
- 续写式状态

### 4.6 训练难度差异

当前主算法：

- 更符合任务
- 但训练难度明显更高
- 更容易出现“loss 在降，但自由生成仍然空”的问题

`v1.0`：

- 更容易降 loss
- 更容易先跑出 baseline
- 但不一定能承载你全部目标

### 4.7 可解释性与调试差异

当前主算法：

- patch 中间资产丰富
- 能看 `prompt/state/gt/pred/ref`
- 可调试性很强

`v1.0`：

- 结构化序列更紧凑
- 训练目标更简单
- 但对“为什么和路口没对齐”这类问题的直接可视化不如当前主算法自然

## 5. 当前主算法的优势与不足

### 5.1 优势

#### 1. 更贴近真实任务

当前主算法直接围绕你真实需要的任务设计：

- `tif`
- `Lane.geojson`
- `Intersection.geojson`

而不是先把问题收缩成另一个更简单的问题。

#### 2. 能同时表达线、面和关系

这套算法天然可以表达：

- 路怎么走
- 路口边界是什么
- 路和路口怎么接

这是它最大的价值。

#### 3. UV patch-local 表示更适合 patch 学习

UV 表示让模型面对的是局部、稳定的坐标空间，对 patch 级学习更友好。

#### 4. 更有机会学到真正拓扑

尤其是在加入：

- state 连续性
- 同 patch companion reference
- 精确接触约束

之后，这套算法有机会学到比“画一串线”更深的关系。

#### 5. 中间资产和诊断能力强

当前主算法已经把 patch 级调试做得很细：

- 输入图
- GT
- state
- companion ref
- raw 输出
- 解析后输出

这对于长期打磨算法非常重要。

### 5.2 不足

#### 1. 训练更难

GeoJSON 文本输出空间大，teacher forcing 与自由生成之间差距也更大，所以更难稳。

#### 2. 更容易学到“空输出是安全答案”

如果空 `FeatureCollection` 是合法的，而数据里又存在空 patch，模型就容易走保守策略。

#### 3. 任务复杂度高

当前不是只学一类几何，而是同时学：

- line
- polygon
- continuity
- topology

难度自然高很多。

#### 4. 对工程调度要求更高

patch 资产、state、companion、UV 转换、推理排序，这些让主链更强，但也让系统更复杂。

---

## 6. v1.0 的优势与不足

### 6.1 优势

#### 1. 更容易训出 baseline

离散序列目标更窄，模型更容易先学会“怎么输出一个像样的序列”。

#### 2. 训练目标更稳定

结构化 token 序列比自由文本 GeoJSON 更容易做标准 next-token 学习。

#### 3. 更适合做快速对照实验

如果你想回答的是：

- “这套数据至少能不能学出线”

那 `v1.0` 很适合拿来做验证。

#### 4. 更适合资源紧张时快速出结果

当目标是快速验证思路而不是一次到位完成最终任务时，`v1.0` 更务实。

### 6.2 不足

#### 1. 任务契合度不如当前主算法

它不是从你的完整目标出发设计的，而是更像一个线图序列 baseline。

#### 2. 对 lane/intersection 联合拓扑支持弱

它不天然擅长：

- 线与面的联合生成
- 精确接触关系表达

#### 3. 需要适配器

`v1.0` 不能直接吃你当前的数据，必须先做数据转换。

#### 4. 当前适配版还不是全量任务

目前适配版主要针对 `Lane`，不是对你整套目标的完整复刻。

## 7. 哪一套更可能得到更好的训练效果

这个问题要拆成两个层次回答。

### 7.1 如果问“哪套更容易先训出一个能跑的结果”

答案更偏向：

- `v1.0`

原因是：

- 输出空间更窄
- 训练目标更收缩
- 更容易形成 baseline

### 7.2 如果问“哪套更有机会最终达到你的真实目标”

答案更偏向：

- 当前主算法

原因是：

- 当前主算法和你的任务目标是对齐的
- 它真正试图让模型理解 lane/intersection/patch continuity/topology
- 而不是只学一个更容易的替代问题

所以更准确的判断是：

- `v1.0` 胜在短期稳定性和 baseline 能力
- 当前主算法胜在长期上限和任务契合度

---

## 8. 什么时候应该选哪一套

### 8.1 适合先跑 `v1.0` 的场景

当你的目标是：

- 先验证当前数据能不能学出基本线条
- 快速获得一个 baseline
- 用更简单的目标验证数据质量和基本可训练性

这时候优先跑 `v1.0` 是合理的。

### 8.2 适合继续做当前主算法的场景

当你的目标是：

- 最终真的产出 `Lane.geojson + Intersection.geojson`
- 真正做到路和路口几何对齐
- 让 patch 间连续性可用
- 让模型自己理解，而不是靠后处理修图

这时候当前主算法才是应该继续投入的主线。

---

## 9. 对当前路线的实际建议

从算法方向上看，我的判断是：

- `v1.0` 值得跑，作为 baseline 和 sanity check
- 但不应该替代当前主算法成为最终路线

更合适的做法是：

1. 用 `v1.0` 验证数据和基础可学习性
2. 用当前主算法继续推进最终目标
3. 把两边结果作为对照，判断问题到底出在：
   - 数据
   - 表示
   - 还是训练难度

这样做的价值最大。

---

## 10. 最终结论

一句话总结：

- `v1.0` 是更容易训稳的线图序列 baseline
- 当前主算法是更贴合你真实目标、但训练难度更高的联合几何生成算法

如果只追求“尽快出一个像样结果”，`v1.0` 更稳。

如果追求“最后真正得到你要的 lane/intersection/topology 效果”，当前主算法更值得长期做。

因此，两套算法不是谁完全替代谁，而是：

- `v1.0` 负责给出对照和 baseline
- 当前主算法负责承载最终任务目标
