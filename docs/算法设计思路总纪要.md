# 算法设计思路总纪要

本文整理这几天围绕 `GeoTIFF -> Lane/Intersection GeoJSON` 训练与推理链路形成的主要算法设计思路，分为两部分：

- 用户明确提出并持续强调的设计目标
- 为了把这些目标真正落到当前代码中，我补充采用的实现型设计决策

文中尽量只写已经进入当前工程主线、或者直接影响当前工程判断的重要思路。对于已经被替换掉的旧路线，会单独标注为“历史阶段”。

## 一、用户明确提出的核心设计目标

### 1. 输入数据与任务定义

用户明确要求训练输入来自真实地理数据，而不是普通自然图像样本。训练样本由以下文件构成：

- `patch_tif/0.tif`
- `patch_tif/0_edit_poly.tif`
- `label_check_crop/Lane.geojson`
- `label_check_crop/Intersection.geojson`

对应任务目标是：

- 从遥感影像生成 `Lane.geojson`
- 从遥感影像生成 `Intersection.geojson`

其中：

- `Lane` 使用 `LineString`
- `Intersection` 使用 `Polygon`
- 属性字段需要一起预测，不只是几何

### 2. 监督范围必须受 review mask 控制

用户很早就明确了一个关键前提：

- `0_edit_poly.tif` 的白色区域才是经过人工审核、可用于监督的区域
- 黑色区域不应作为训练监督来源

这直接决定了后续整条链必须围绕“按 mask 裁图、按 mask 裁真值”展开，而不是简单整图监督。

### 3. 大图必须支持 patch 级训练和 patch 级推理

用户明确指出：

- 输入 tif 很大
- 不可能直接整图送进模型

因此算法上必须支持：

- 训练期大图切 patch
- 推理期大图滑窗推理
- patch 结果再拼回整图

这不是工程细节，而是主算法路线的一部分。

### 4. 模型总体形态：冻结 DINO + 模态转换器 + Qwen

用户反复确认并接受的总体模型结构是：

- 用 DINO 做遥感图像视觉编码
- 用一个视觉到语言的模态转换器把视觉 token 对齐到 Qwen hidden size
- 用 Qwen 负责自回归生成地图文本

同时用户明确要求：

- DINO 冻结
- 需要支持两种训练模式
  - projector + Qwen LoRA
  - projector + Qwen full fine-tune

### 5. patch 之间需要存在连续性，而不是完全独立

用户多次强调，不希望每个 patch 完全无关地独立预测，而是希望 patch 之间存在承接关系。这个需求后来收敛成两类设计目标：

- patch 顺序要可控
- 相邻 patch 的断点信息要能够传给后续 patch

这就是后续 `state / cut / anchor` 设计的来源。

### 6. 用户希望能看到每一张 patch 的训练和推理中间资产

用户明确要求不仅看最终整图输出，还要看：

- 每张 patch 的输入图
- 每张 patch 的 GT
- 每张 patch 的 state/cut
- 每张 patch 的预测结果
- 以及对应原始文本

这促成了后来“全 patch 中间资产导出”和“tile 级 debug 文件”的设计。

### 7. 文本输入输出不要使用经纬度，而要使用 patch-local UV

这是近期最重要的一次设计转向。用户明确要求：

- 大模型全程不要直接处理经纬度
- prompt 中的坐标、监督文本中的坐标、state 中的坐标，都改成 UV
- 最终输出和中间资产再把 UV 转回实际地理坐标

这个要求的本质是：

- 让模型在局部稳定坐标空间里学习
- 避免经纬度数值范围大、冗长、分布不稳

### 8. `CRS / Id / RoadId` 不应该进入训练 loss

用户明确指出这些字段的性质：

- `crs.name` 是固定值
- `Id` 与 `RoadId` 原始数据里往往是随机或不稳定编号

因此不应该让模型去“背编号”，而应该：

- 把它们从训练文本中剥离
- 在解码后统一重建

### 9. 编号规则需要稳定、可控、后处理统一完成

在不把 `Id/RoadId` 放进 loss 之后，用户又进一步提出了确定性编号规则：

- `Id` 在 `lane + intersection` 间共用一条全局递增序列
- `RoadId` 只对 `lane` 使用，单独递增

这意味着编号必须是：

- 后处理规则
- 不是模型自由生成结果

### 10. 需要强可观测性与可排障性

用户持续要求：

- 能看 prompt 到底是什么
- 能知道模型卡在哪一步
- 能知道是哪个 patch 卡住
- 能知道输出为空是 GT 空、state 空、模型空，还是后处理删空

这直接推动了后续一整套诊断设计，而不是只看最终 loss。

## 二、为落地这些目标而做出的关键实现决策

下面这些不是用户最初直接用同样术语提出的，但都是为了把上面的目标在现有代码框架中真正做成，而我在实现过程中决定采用的方案。

## 1. patch 连续性采用“状态压缩传递”，而不是完整多轮问答

用户曾经讨论过“连续问答式”训练，但当前落地没有采用完整 chat-history 方案，而是采用了：

- `prompt_text`
- `state_text`
- `target_text`

这种三段式结构。

核心思路是：

- `prompt_text` 负责任务说明和局部上下文
- `state_text` 负责传递相邻 patch 的 cut/anchor 先验
- `target_text` 只作为当前 patch 的监督标签

这是一种“状态条件化单轮训练”，而不是真正多轮聊天历史累积。原因是：

- 更省上下文长度
- 更适合 16G 显卡
- 更容易稳定训练

## 2. 训练采用 teacher forcing，而不是自由生成训练

当前训练前向里，模型看到的是：

- 视觉前缀
- prompt
- state
- 真值 target token 前缀

训练目标是 next-token 交叉熵。

这意味着：

- 训练 loss 下降，只说明 teacher forcing 条件下 token 预测变好
- 不等于自由生成时一定能稳定输出非空对象

这个设计是标准 LLM 训练范式，优点是：

- 实现成熟
- 训练稳定
- 容易和 Qwen Causal LM 对齐

缺点也很明确：

- 推理空输出问题不会直接被 loss 完整惩罚

## 3. 当前主链选择“UV GeoJSON 文本”而不是老的纯结构化离散 token 输出

历史上工程里保留过更强结构约束的 tokenizer/grammar 路线，但当前主线已经收敛到：

- 用 `GeoCoordTokenizer` 负责文本编码解码
- 训练监督文本是 UV GeoJSON 文本
- 推理输出先解为 UV GeoJSON，再转回像素/世界坐标

这样做的原因是：

- 比自由自然语言更可控
- 比完全离散结构 token 更容易快速落地与调试
- 仍然能保留几何的显式文本结构

可以把当前方案理解为：

- 结构化文本主链
- 而不是完全自由文本链
- 也不是最老的全离散几何 token 主链

## 4. 训练文本显式剥离无效字段

为了满足“`CRS / Id / RoadId` 不进入 loss”这一要求，当前实现里采用的是：

- 在 dataset 阶段，先构造 UV GeoJSON
- 再调用 `strip_non_training_fields_from_feature_collection(...)`
- 得到真正进入 `state_text/target_text` 的训练文本

这是一种比“在 prompt 里让模型忽略这些字段”更强的实现方式。它的优势是：

- 不依赖模型理解
- 直接从训练数据层面剥掉
- 可确保 loss 不再覆盖这些字段

## 5. 编号改成解码后统一赋值

为了满足用户提出的稳定编号规则，当前实现采用：

- 模型不学习 `Id/RoadId`
- 最终 patch 或整图结果在导出前统一调用 `assign_incremental_feature_ids(...)`

这是一种明确的“生成与编号解耦”设计。它的意义是：

- 模型只负责几何与稳定属性
- 编号由后处理保证确定性和一致性

## 6. patch 审计和推理调试采用“双轨输出”

为了让“哪一块 patch 出问题”能被追出来，当前实现用了双轨调试：

### patch 审计轨

用于记录：

- 候选 patch
- 是否被选中
- bbox / keep_bbox
- mask_ratio / mask_pixels

### tile debug 轨

用于记录：

- 当前 tile 的 prompt_text
- state_text
- state_anchor_count
- prompt/state token 数
- max/min new tokens
- generate / parse / keep 用时
- 原始预测文本

这套设计的目的不是为了漂亮，而是为了把“空输出”拆成：

- GT 空
- state 空
- 模型空
- 解析失败
- keep_bbox 丢光

## 7. 即使 `tile_audit` 原始记录为空，也增加 fallback 审计

用户多次遇到：

- `tile_audit.json` 是空
- 但又想知道 patch 到底怎么切的

因此当前实现里，我增加了 fallback 设计：

- 如果预测侧 `tile_audit` 为空
- 就直接从 `tile_windows` 反推出一套可导出的 patch 审计记录

后面训练侧也补了类似 fallback。

这是一个典型“为了可诊断性而补的工程算法决策”。

## 8. 输出语义做了分层

为了避免把“模型原始输出”和“后处理规范化结果”混在一起，当前链路实际上已经形成了三层语义：

- `*.pred.raw.txt`
  - 模型原始文本
- `*.pred.uv.geojson`
  - 从原始文本中提取出来的合法 UV GeoJSON
- `*.pred.geojson`
  - 再经过坐标恢复和标准化后的结果

这套分层非常重要，因为它帮助区分：

- 模型是否真的生成了完整对象
- 还是后处理把一个半残结构“修正”成了合法 GeoJSON

## 9. 针对 16G 显卡，整体设计选择了“收缩型主线”

为了让项目在 16G 上勉强跑通，当前实现里做了一套系统性收缩，而不是只调一个参数：

- 冻结 DINO
- Qwen 走 LoRA
- 启用 gradient checkpointing
- 半精度训练
- `sat_token_hw` 收到 `[2, 2]`
- `image_size` 收到 `256`
- `tile_size_px` 收到 `768`
- `batch_size=1`
- `prompt/state/target` 限长
- `max_new_tokens` 显式收紧

这些已经单独整理在 [16G显存适配说明](/c:/DevelopProject/VScode/geomapgen/docs/16G显存适配说明.md) 里。

## 10. 训练与调试模式被明确区分

最近排查空输出时，我们故意把一些“长期不该默认打开”的开关重新打开了，例如：

- 训练期保存预测
- 每个 batch 都导预测
- 导出所有 patch 图

这不是因为这是最佳训练算法，而是因为：

- 当前阶段目标从“高吞吐训练”切换成了“定位空预测根因”

因此当前工程实际上存在两套思路：

### 稳定训练思路

- 少导资产
- 只看验证预测
- 开 cache
- 开 worker

### 深度排障思路

- 全 patch 导出
- 训练期也导预测
- 加 tile 级 debug
- 强化日志

## 三、这些天形成的设计演化脉络

为了后续看懂为什么代码会长成现在这样，可以把这几天的主线变化概括成下面几步。

### 阶段 1：从“整图 / 普通文本”问题，转到“patch + 受 mask 监督”的真实地理任务

这一步确定了：

- 必须滑窗
- 必须只在审核区域监督
- 必须支持真实 CRS / tif / geojson

### 阶段 2：从“完全独立 patch”转到“patch 之间有 state 承接”

这一步确定了：

- 需要 `state_text`
- 需要 cut/anchor 先验
- 需要顺序化 patch 训练和推理

### 阶段 3：从“世界坐标文本”转到“UV 文本主链”

这一步确定了：

- 大模型全程只看 UV
- 世界坐标只用于导出
- prompt/state/target 全部统一到 UV 空间

### 阶段 4：从“模型学习编号和 CRS”转到“后处理统一赋值”

这一步确定了：

- `CRS / Id / RoadId` 不参与训练
- 这些字段由解码后规则系统补回

### 阶段 5：从“只看 loss”转到“必须看 patch 级中间资产和日志”

这一步确定了：

- 需要全 patch 资产
- 需要 tile debug
- 需要分阶段推理日志
- 需要空输出专门诊断文件

## 四、当前最重要的认识

经过这几天的反复排查，当前已经形成几个非常关键的共识。

### 1. loss 下降不等于自由生成成功

当前训练使用 teacher forcing，因此：

- loss 下降只说明 token 级局部预测变好
- 不说明推理时一定能稳定生成非空 GeoJSON

### 2. “空 FeatureCollection”是一个低风险合法答案

这会导致：

- 模型在训练早期很容易走空输出路径
- 只看 loss 容易产生错觉

### 3. `raw.txt`、`pred.uv.geojson`、`pred.geojson` 必须分开看

否则很容易误以为：

- 模型写得很完整

实际上可能只是：

- 后处理抽取、包裹、补默认值、坐标恢复之后看起来完整

### 4. patch 审计、state 审计、生成调试必须长期保留一部分

哪怕后面不再全量导出，也应该保留一套最小诊断路径。因为这个任务不是普通分类任务，而是：

- patch 级滑窗
- 状态承接
- 几何裁剪
- 文本生成
- GeoJSON 解析
- 坐标恢复

任何一层都可能把结果变空。
