# 16G显存适配说明

本文记录当前项目为了在 16G 显卡上尽量跑通训练与推理所做的主要优化，以及哪些调试开关会暂时抵消这些优化。

## 目标

当前项目的主链是：

- `DINO` 遥感编码器
- `DINO -> Qwen` 的轻量桥接层 `sat_proj`
- `Qwen` 语言模型

对 16G 显卡来说，真正的风险不只是图像，而是：

- LLM 激活占用
- prompt/state/target 文本长度
- 训练期预测导出
- patch 级大量裁图与写盘

所以适配策略不是单点优化，而是一整套“模型收缩 + 文本收缩 + patch 收缩 + 导出收缩”。

## 一、模型侧优化

当前配置见 [geo_vector_lora.yaml](/c:/DevelopProject/VScode/geomapgen/configs/geo_vector_lora.yaml)。

### 1. 冻结遥感编码器

- `model.freeze_satellite: true`

作用：

- 不训练 DINO 本体
- 只保留 DINO 前向提特征
- 显著减少梯度和激活保存开销

### 2. Qwen 走 LoRA，不做全参训练

- `model.llm_train_mode: lora`

LoRA 细节：

- `r: 4`
- `alpha: 8`
- `dropout: 0.05`
- `target_modules: [q_proj, v_proj]`

作用：

- 不更新整个 Qwen
- 只训练少量 LoRA 参数和桥接层
- 这是 16G 能勉强训练的核心前提

### 3. 启用梯度检查点

- `model.gradient_checkpointing: true`

作用：

- 用重算换显存
- 训练会变慢，但显存明显更稳

### 4. LLM 使用半精度

- `model.llm_torch_dtype: float16`
- `train.amp: true`

作用：

- LLM 参数和训练过程使用半精度
- 进一步压低显存占用

### 5. 压低视觉前缀长度

- `model.sat_token_hw: [2, 2]`

作用：

- DINO 输出只保留很少量的视觉 token
- 减少进入 Qwen 的前缀长度
- 直接减少后续注意力和序列开销

## 二、输入尺寸与 patch 侧优化

### 1. 缩小模型输入图尺寸

- `data.image_size: 256`

作用：

- patch 裁出来后只 resize 到 `256x256`
- DINO 前向更轻
- 显存和算力压力都更低

### 2. 限制 patch 大小

训练和预测默认都收到了：

- `tile_size_px: 768`
- `overlap_px: 192`
- `keep_margin_px: 96`

作用：

- 避免单个 patch 过大
- 控制每个 tile 的图像内容和几何复杂度
- 减少单次生成的上下文压力

### 3. 训练批大小固定为 1

- `train.batch_size: 1`
- `train.val_batch_size: 1`

作用：

- 这是 16G 下最保守也最现实的选择
- 当前项目不适合在 16G 上做更大的 batch

### 4. 单样本逐 patch 训练

- `epoch_is_single_sample: true`
- `sample_patch_sequential: true`
- `val_sample_patch_sequential: true`

作用：

- 每轮只训练一张大图的全部 patch
- 避免同时混很多 sample 带来的上下文和导出复杂度

## 三、文本与监督侧优化

### 1. 明确收紧 prompt / state / target token 长度

当前默认值：

- `prompt_max_tokens: 96`
- `state_max_tokens: 96`
- `target_max_tokens: 256`

这部分是 16G 适配里非常关键的一步。

如果这里放开：

- prompt 太长
- state 太长
- target 太长

那么显存主要会爆在 Qwen 的长序列上，而不是图像上。

### 2. prompt 模板做了收缩

当前模板只保留：

- patch 内任务说明
- 最少必要 few-shot 示例
- 不生成 `CRS / Id / RoadId` 的约束

作用：

- 减少无关 token
- 降低 prompt 长度

### 3. 训练文本不再包含无关字段

训练文本中已经移除：

- `crs`
- `Id`
- `RoadId`

对应代码：

- [dataset.py](/c:/DevelopProject/VScode/geomapgen/unimapgen/geo/dataset.py)
- [io.py](/c:/DevelopProject/VScode/geomapgen/unimapgen/geo/io.py)
- [prompting.py](/c:/DevelopProject/VScode/geomapgen/unimapgen/geo/prompting.py)

作用：

- 减少模型要背的无效字段
- 缩短 target 文本
- 避免把随机编号浪费进 loss

### 4. 使用 UV 文本而不是经纬度文本

当前主链已经改成：

- 模型输入文本使用 patch-local UV
- 最终导出时再转回世界坐标

作用：

- UV 比经纬度短、更稳定
- 文本 token 更紧凑
- 对 16G 更友好

## 四、几何复杂度侧优化

### 1. 限制每类 feature 数量

当前默认值：

- `lane.max_features: 32`
- `intersection.max_features: 16`
- `state_update.max_features: 4`
- `state_anchor_max_points: 3`

作用：

- 控制单 patch 的监督复杂度
- 减少 target/state 文本体积
- 防止某些复杂 patch 直接把序列长度顶爆

### 2. 采样间隔保持在较保守水平

- `sample_interval_meter: 6.0`

作用：

- 降低单条线的点密度
- 减少 target 中坐标点数量

### 3. coord bins 收紧

- `coord_bins: 640`

虽然当前主链是 UV 文本，不是老的纯离散坐标 token 输出，但这个值仍然保持在较轻水平，避免序列和坐标空间继续膨胀。

## 五、训练前向和推理侧优化

### 1. 训练主循环默认不保留 logits

训练前向调用时使用：

- `return_logits=False`

作用：

- 不把完整 logits 留到外层
- 减少训练过程中的额外显存占用

### 2. 训练前向禁用 KV cache

在 `QwenSatelliteMapGenerator.forward()` 内部：

- `self.llm(..., use_cache=False, ...)`

作用：

- teacher forcing 训练不保留 KV cache
- 减少训练态显存

### 3. 预测生成长度收紧

预测脚本当前默认：

- `max_new_tokens: 64`
- `min_new_tokens: 16`
- `top_k: 1`

训练脚本当前默认：

- `max_new_tokens: 128`
- `min_new_tokens: 16`

作用：

- 避免生成长度失控
- 防止推理阶段出现“接近上下文上限”的超长解码

### 4. 预测参数解析修复

之前如果 `max_new_tokens=0`，代码会退回到接近上下文上限，导致极慢且很重。

现在在 [predict_geo_vector.py](/c:/DevelopProject/VScode/geomapgen/unimapgen/predict_geo_vector.py) 里已经修复成：

- 命令行正数优先
- 否则用配置正数
- 再否则回退安全默认值

这一步对 16G 的预测稳定性非常关键。

## 六、内存风险评估

训练入口里已经有一套显存风险估计：

- [train_geo_model.py](/c:/DevelopProject/VScode/geomapgen/unimapgen/train_geo_model.py)
  - `_estimate_memory_risk(...)`

这套规则会重点盯这些高风险项：

- `llm_train_mode=full`
- `image_size` 过大
- `target_max_tokens` 无限制或过大
- `state_max_tokens` 无限制或过大
- `lane/intersection.max_features` 无限制
- `batch_size > 1`
- `amp=false`
- `gradient_checkpointing=false`

这相当于把 16G 适配经验编码进了训练初始化检查。

## 七、当前哪些设置已经不是“16G优先”，而是“调试优先”

下面这些是为了最近排查空预测和空审计打开的重型设置，它们会明显拖慢训练，不属于长期高吞吐配置：

- `artifact_export.save_train_batch_predictions: true`
- `artifact_export.max_batches_per_epoch: 0`
- `artifact_export.max_patch_images_per_sample: 0`
- `cache_enabled: false`
- `num_workers: 0`
- `val_num_workers: 0`

它们的影响是：

- 训练中每个 batch 都会额外跑预测生成
- 每个 patch 都会写图、写文本、写 GeoJSON
- 数据预处理全在主线程完成

所以你最近看到：

- GPU 占用不高
- 显存只在 40% 到 70%
- 训练卡在 `0%`

这并不表示 16G 还有很多余量，而是说明瓶颈被这些调试导出和 CPU/IO 工作拿走了。

## 八、如果要切回“更适合 16G 连续训练”的模式

建议把这些值改回保守版：

### 训练期预测导出

- `save_train_batch_predictions: false`
- `max_batches_per_epoch: 1`
- `max_patch_images_per_sample: 1`

### 数据读取

- `cache_enabled: true`
- `num_workers: 2`
- `val_num_workers: 2`

这样会更接近真正的“16G 可持续训练模式”。

## 九、结论

当前项目为了适配 16G，核心不是某一个单点优化，而是下面这一组组合拳：

- 冻结 DINO
- Qwen 改成 LoRA
- 开启半精度和梯度检查点
- 压低视觉 token 数
- 把图像输入缩到 `256`
- patch 缩到 `768`
- batch size 固定为 `1`
- prompt/state/target 严格限长
- 去掉 `CRS / Id / RoadId` 这类无关训练字段
- 预测生成长度显式收紧

如果这些设置被放开，16G 很快就会从“能跑”退回到“很难稳定训练”。

当前仓库里为了排查问题，又额外打开了一些重型 debug 导出，所以最近看到的训练速度和 GPU 利用率，不应被当成 16G 适配的上限表现。
